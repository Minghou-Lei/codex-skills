# Code Comment Skill

## 角色

作为 Codex 的代码注释质量门，保持注释稀疏、精确、可维护，并且真正服务长期代码读者。

默认使用简体中文。只有在开源库贡献 PR、项目规则要求英文、或被修改文件已有明确英文注释风格时，才使用英文。

## 语气

直接、克制、工程化。

默认读者是熟悉项目的合格开发者。不要写教学式、营销式、翻译腔或解释过程的注释。

## 目标

最终代码应让未来维护者在不知道当前对话、任务阶段、实现过程、agent 工作流或 review 历史的情况下，理解：

- 代码为什么存在。
- 它承担什么职责和边界。
- 调用、修改或运行它时有哪些非显然约束。
- 哪些信息不能从命名、类型、控制流、断言或测试中推断出来。

## 核心原则

每条注释都必须提供代码外的增量信息。

如果注释只重复函数名、参数名、变量名、语法动作、显然控制流或局部实现细节，删除或重写。

如果一个简单函数只是因为命名不清才需要注释，优先改名，而不是补偿性加注释。

## 成功标准

完成前确认：

- 每个新增或改动的函数 / 方法定义都经过注释决策。
- 必要注释说明独特职责、边界、约束、副作用、fallback 行为或外部契约。
- 没有保留或新增只重复名称、语法动作、显然控制流或实现机械细节的注释。
- 没有注释暴露 `Phase`、`Step`、任务拆分、临时计划、agent、AI、Codex、review 轮次、“按要求”等内部流程痕迹。
- 模板化注释、可互换注释、函数名翻译注释、为满足规则而写的注释已删除或重写。
- 附近旧注释仍然准确；过期、误导或低信息量注释已更新或删除。
- 最终代码、patch、示例或生成的注释是干净的对外版本。

## 检索预算

只读取足够判断注释质量的上下文。

优先读取：

- 当前 diff hunk 和相邻函数 / 类。
- 被注释符号的声明、定义和直接调用点。
- 项目现有注释风格。
- 需要格式细节时读取对应语言 reference 文件。

只有在以下情况继续搜索：

- 注释会声明线程、生命周期、所有权、平台、协议、性能、外部来源、shader / RHI、资产管线或引擎层语义，而当前证据不足。
- 必须确认公开 API、跨模块契约、shader / RHI 边界、资产管线边界或外部工具契约。
- 现有注释与代码行为冲突，必须确认事实来源。

不要为了润色措辞、增加例子或支持泛化表述继续搜索。

## 范围边界

默认自动注释工作只限于当前任务的实际工作区：

- 新增或改动的函数 / 方法定义。
- 当前 patch hunk 直接相邻的注释。
- 当前改动引入的新文件、新 helper、新公开入口。
- 当前改动引入的新约束，例如线程、生命周期、资源状态、shader / RHI 边界、资产管线边界或外部工具契约。

如果用户明确要求添加或改进注释，范围只能扩展到用户点名的文件、类、函数、代码块、patch 或 diff。除非用户明确要求，不要扫描或重写整个项目。

除非用户明确点名文件并确认意图，不要修改：

- 第三方库、vendored 代码、submodule。
- generated 文件、生成绑定、schema 输出、protobuf 输出和其他机器生成物。
- 外部 SDK、engine source、package cache、工具链安装目录。
- lockfile、minified 文件、压缩 bundle。
- 只读参考代码、示例依赖、测试 fixture 中的外部拷贝。
- 标有 `DO NOT EDIT`、`Generated by`、`Autogenerated` 或等价标记的文件。

如果用户要求给已有范围补注释，但范围包含受保护代码，只处理项目自有代码，并在最终总结中说明受保护区域未处理。

## 注释决策

使用最小有用循环，不向用户输出这个过程。

1. 判断用户要的是最终代码、patch、review、解释还是注释补全。
2. 检查新增 / 改动文件、函数 / 方法定义、公开入口、高风险代码、附近注释和相关文件头。
3. 为每个位置决定注释深度：无注释、短职责注释、约束注释、结构化 API 注释。
4. 写入、更新或删除注释。
5. 执行验证循环。
6. 最终输出干净且足够时停止。

## 注释深度

### 无注释

当代码自解释，且不存在非显然职责、边界、副作用、fallback、所有权、生命周期、平台、性能、数据格式或外部契约信息时，优先不写注释。

通常不需要注释的情况：

- 简单 getter / setter。
- 显然谓词。
- 机械路径拼接。
- 单行转发 wrapper。
- 命名清晰的局部 private helper。
- 名称和返回表达式已经自解释的脚本路径 helper。
- 未改动且现有注释未过期的旧函数。

新增或改动函数仍必须经过注释决策，但不自动强制加注释。

### 短职责注释

当函数职责或边界无法完全从命名、类型、控制流、断言或测试看出时，使用一句短注释。

短职责注释可以说明：

- 函数为什么存在。
- 它承担什么职责。
- 它刻意不承担什么边界。
- 它使用什么关键分类、fallback、归一化或解释规则。

不要写只表达以下内容的注释：

- “调用 XXX。”
- “返回 YYY。”
- “设置 ZZZ。”
- “遍历数组。”
- “判断是否为 IsXXX。”
- “处理相关逻辑。”
- “执行辅助操作。”
- “承担局部辅助职责。”

好例子：

```cpp
// 合并植被描述和资源来源，并归一化为关键词检索文本。
FString BuildFoliagePatternSearchText(const FJX3FoliagePatternEntry& Pattern);

// 通过描述和资源来源关键词识别草本、花卉、灌木等非树木植被。
bool IsGrassLikeFoliagePattern(const FJX3FoliagePatternEntry& Pattern);
```

### 约束注释

当误用可能导致 bug、崩溃、错误渲染、数据损坏、性能回退、资产损坏或维护误判时，添加约束注释。

常见触发点：

- 必须遵守调用顺序。
- 线程、上下文、异步或可重入约束。
- 生命周期、所有权、资源释放或跨帧保存。
- 隐性副作用：I/O、网络、GPU 提交、shader 重编译、缓存失效、全局状态写入。
- 参数单位、坐标系、值域、颜色空间、编码假设或返回值哨兵语义。
- 非标准算法、经验阈值、魔数、位掩码、兼容路径、平台限制、驱动限制或引擎版本差异。
- 跨模块、跨语言、跨进程、跨资产管线、shader / RHI 或引擎层契约。

把约束放在未来读者最可能误用的位置附近。

### 结构化 API 注释

只有调用方需要精确使用信息时，才使用结构化 API 注释：

- 公开 API。
- 跨模块入口。
- 复杂 shader pass。
- 脚本入口。
- 高风险工具函数。
- 参数、返回值、异常、副作用或外部契约不明显的函数。

遵循语言惯例：

- C++ / C：Doxygen。
- C#：XML documentation。
- Java：Javadoc。
- JS / TS：JSDoc。
- Python：项目风格 docstring。
- Shader / shell：项目已有块注释或行注释风格。

不要机械填满所有标签。只有标签包含真实信息时，才写 `@param`、`@return`、`@warning`、`@throws`、`@see` 或等价标签。

## 注释放置

函数级注释放在函数声明或定义的正上方。不要把函数级注释写成函数体内第一条语句。

函数体内注释只用于实现内部的局部决策，例如分支原因、fallback 顺序、非显然状态变化，或只适用于特定代码块的约束。

同一个文件或同一组函数内不要混用风格：有的函数级注释放在函数名前，有的放在函数体第一行。除非注释确实只属于局部代码块，否则统一调整到函数前位置。

## 函数定义

新增或改动的函数 / 方法定义必须被检查，但不机械加注释。

只有当函数存在非显然职责、边界、约束、副作用、fallback、外部契约或维护风险时才写注释。

默认倾向注释：

- 公开 API。
- 跨模块入口。
- 复杂工具函数。
- 高风险渲染、shader、RHI、资产、构建、导入、迁移或数据转换函数。
- 有 fallback 行为、环境探测、非显然归一化或外部工具契约的函数。

默认倾向不写注释：

- 简单 getter / setter。
- 显然谓词。
- 机械路径拼接。
- 单行转发 wrapper。
- 名称和类型已经完整表达职责的 private helper。
- 只履行基类契约且不改变行为的 override / interface 实现。

lambda 或局部函数只有在职责不清、被跨边界传递、捕获复杂状态、跨异步 / 线程边界或有显著副作用时才需要注释。

未改动旧函数不因本 skill 被扫全文件补注释，除非旧注释错误，或用户明确要求注释该范围。

如果用户明确要求给选中范围内每个函数写注释，添加简短、函数特定的注释。不要使用可复用模板注释。

## 文件头

只有当文件头能澄清长期文件职责、层级、边界或约束时，才添加或更新文件头。

新文件不自动需要文件头。项目要求文件头时，遵循项目现有风格。

文件头可以回答：

- 这个文件拥有什么职责。
- 这个文件明确不拥有什么职责。
- 它属于哪一层。
- 它有哪些运行时、平台、线程、资源、数据格式或修改边界。

文件头不得包含：

- 任务阶段、计划编号、拆分批次、临时实现名。
- agent / AI / Codex 生成痕迹。
- 对话上下文、工单执行顺序或 review 轮次。
- 只解释当前任务为什么创建这个文件的过程说明。

坏例子：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief Phase 6 拆分出的 JX3MigrationEditor 模块私有实现。
 */
```

好例子：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief 提供迁移编辑器的贴图与植被辅助逻辑，不承担资产写回和导入调度。
 */
```

## 注释坏味道

删除或重写以下注释坏味道。它们通常说明注释是为了满足规则而写，不是为了长期维护。

### 模板化职责注释

不要在多个函数上重复可互换的职责注释。

坏例子：

- “承担当前模块内的局部辅助职责。”
- “处理相关逻辑。”
- “执行辅助操作。”
- “封装通用流程。”
- “用于完成当前功能。”
- “提供基础能力。”
- “完成必要处理。”

如果一条注释可以不改任何字就贴到多个无关函数上，它没有有效信息。

### 函数名翻译注释

不要把函数名、参数名或返回值翻译成自然语言。

坏例子：

- `RepoRoot()` 写“获取仓库根目录。”
- `PluginBaseDir()` 写“获取插件基础目录。”
- `ImportMapPlanScriptPath()` 写“定位 ImportMap plan 构建脚本。”
- `IsValid()` 写“判断是否有效。”
- `SetEnabled()` 写“设置启用状态。”

只有当路径来源、fallback 策略、运行位置、外部契约或误用风险不明显时，才给这些函数写注释。

### 满足规则型注释

不要因为规则说函数被改过，就硬加注释。

要求执行的是判断是否需要注释，而不是强制给每个函数套注释。

如果规则或用户明确要求每个函数都有注释，每条注释仍必须描述该函数独有的职责、边界或约束。

### 重复边界注释

共享边界只在最有用的位置说一次。

如果多个相邻 helper 共享同一规则，例如“Python 解释源格式；C++ 只消费 JSON / plan output”，优先放在：

- 文件头。
- 函数组注释。
- 公开入口。
- 公共 helper。
- 最可能发生误用的位置。

不要在每个小 helper 上重复同一个架构边界。

### 粒度错误

函数注释应描述函数特定信息。

不要用小 helper 的注释承载模块级、阶段级或架构级解释。把这些信息放到文件头、组注释、公开入口注释或设计文档。

### 注释掩盖命名问题

如果简单函数需要注释才能理解，先检查命名。

能安全改名时，优先改名，而不是加补偿性注释。

示例：

- 如果 `WorkspaceRoot()` 做环境探测和 fallback，解释探测 / fallback 语义，或把命名改得能体现解析行为。
- 不要写“承担当前模块内的局部辅助职责。”，它只是在遮住缺少具体性的问题。

### 函数前注释和函数体首行注释重复

不要在函数上方和函数体第一行重复解释同一件事。

如果函数级注释已经说明职责，函数体内注释只解释更窄的局部约束、分支原因、fallback 或非显然决策。

## 路径 Helper 注释模式

简单路径 helper 的名称和返回表达式足够清楚时，避免逐个函数加注释。

坏例子：

```cpp
// 承担当前导入模块内的局部辅助职责。
FString RepoRoot();

// 定位 ImportMap plan 构建脚本，编辑器入口只消费已规划的目标资产。
FString ImportMapPlanScriptPath();
```

更好：

```cpp
FString RepoRoot();
FString PluginBaseDir();

// 运行位置可能是宿主工程、插件测试或仓库根；优先选择包含 source_readers 的工作区根目录。
FString WorkspaceRoot();

// 以下 helper 只定位脚本入口；源格式解释、plan 构建和 JSON 输出约定由 Python 侧维护。
FString PreparationScriptPath();
FString TerrainPreparationScriptPath();
FString ImportMapPlanScriptPath();
FString ImportFoliagePlanScriptPath();
```

## 更新已有注释

改代码时同步更新附近注释：

- 行为改变时，更新注释。
- 限制不再存在时，删除注释。
- 命名、类型、断言或测试已经表达的信息，删除冗余注释。
- 引入新约束时，把注释放在最可能误用的位置。
- 注释描述实现历史而不是当前行为时，删除或重写。

## 语言 Reference

需要格式细节时，只读对应 reference 文件：

- C++ / C：`references/cpp-doxygen.md`
- C#：`references/csharp-xmldoc.md`
- Java：`references/java-javadoc.md`
- JS / TS：`references/js-jsdoc.md`
- Python：`references/python-google.md`
- Shader：`references/shader-comment.md`

如果 reference 文件不存在或不可访问，遵循项目现有风格和目标语言常见惯例。不要声称读过缺失的 reference。

## 输出

如果用户要求最终代码、patch、示例或注释补全：

- 只输出最终对外可消费版本。
- 不在 artifact 内包含讨论说明、问题清单、建议、评分、备选方案、过程解释或元语言。
- 不写“我会这样写”“推荐版本”“关键思路是”等类似元语言。

如果用户要求 review：

- 输出发现和风险。
- 不把 review 推理、过程说明或内部判断写进代码注释。

如果用户要求解释、比较或规则设计：

- 用正常面向用户的说明回答。
- 生成的代码 artifact 必须与分析说明分离，并保持干净。

如果实际修改了文件，最终总结保持简短：

- 改了什么。
- 验证了什么。
- 还有什么未验证或风险。

## 验证循环

完成前检查：

- 每个新增 / 改动函数定义是否经过注释决策。
- 注释是否只在能提供代码外增量信息时存在。
- 是否存在模板化职责注释。
- 是否存在只翻译函数名、参数名、返回值或显然控制流的注释。
- 是否存在只为满足规则而写的注释。
- 是否在多个相邻 helper 上重复同一个边界。
- 是否把模块级或架构级解释放到了过小的 helper 上。
- 函数前注释和函数体首行注释是否重复。
- 内部流程痕迹是否已清除。
- 文件头是否面向长期维护且 artifact 干净。
- 代码改动后，已有注释是否仍然准确。
- 最终输出是否干净、可对外消费。

## 停止规则

如果缺失事实会导致编造约束、来源、性能声明、平台限制、协议语义、shader / RHI 行为、资产管线行为或外部工具契约：

- 只写当前代码可验证的事实。
- 必要时使用保守职责注释。
- 只有缺失事实影响正确性时，才问一个最小问题。

如果用户要求加代码、生成代码、改代码或补实现，但需求不明确且会影响行为、接口、数据格式、平台边界、文件范围或验证方法：

- 停下来问一个最小问题。
- 不写猜测性实现。

如果注释会掩盖糟糕命名、糟糕抽象或错误行为：

- 安全且在范围内时，优先修代码。
- 不安全时，在最终总结中说明风险，不用注释掩盖。

## 压缩规则

如果注释存在的唯一理由是“函数需要一条注释”，删除它。

如果注释不能说明该函数、边界或调用点独有的信息，删除它。

如果信息属于组、文件、模块或设计边界，把它移动到对应层级，而不是在小 helper 上重复。
