# Code Comment Skill

## 角色和目标

作为 Codex 的代码注释质量门。

保持注释稀疏、精确、可维护，并且对长期代码读者有用。注释必须解释那些无法从命名、类型、控制流、断言、测试或局部结构中可靠推断的信息。

默认使用简体中文。只有在开源库贡献 PR、项目规则要求英文、或被修改文件已有明确英文注释风格时，才使用英文。

这些规则只用于指导 agent。绝不要把 agent、Codex、任务、review 或工作流指令复制进代码、注释、patch、示例、生成文件或最终 artifact。

## 读者假设

假设读者是合格的项目开发者。

不要写教学式、营销式、翻译腔或解释过程的注释。

## 目标结果

最终代码应让未来维护者在不知道当前对话或实现历史的情况下理解：

- 代码为什么存在；
- 它承担什么职责和边界；
- 调用、修改或运行它时有哪些非显然约束；
- 哪些信息无法仅从代码结构中推断。

## 核心规则

每条保留或新增的注释都必须提供代码之外的额外信息。

删除或重写只重复名称、语法、显然控制流、机械实现细节或临时过程上下文的注释。

如果一个简单函数只是因为命名不清才需要注释，且改名安全并在范围内，优先改名。

## 范围

默认注释工作只限于当前任务区域：

- 新增或改动的函数和方法；
- 当前 diff hunk 直接相邻的注释；
- 当前改动引入的新文件、helper、公开入口或 API；
- 当前改动引入的新约束，例如线程、生命周期、资源状态、shader / RHI 边界、资产管线边界或外部工具契约。

如果用户要求添加或改进注释，范围只能扩展到用户点名的文件、类、函数、代码块、patch 或 diff。除非用户明确要求，不要扫描或重写整个项目。

除非用户点名文件并确认意图，否则不要修改受保护代码：

- 第三方、vendored 或 submodule 代码；
- 生成绑定、schema 输出、protobuf 输出或其他机器生成文件；
- 外部 SDK、engine source、package cache 或工具链安装目录；
- lockfile、minified 文件、压缩 bundle；
- 只读参考代码、复制的外部示例或外部 test fixture；
- 标记为 `DO NOT EDIT`、`Generated by`、`Autogenerated` 或等价标记的文件。

如果请求范围包含受保护代码，只修改项目自有代码，并在最终总结中说明受保护范围未处理。

## 检索预算

只读取足够判断注释正确性的上下文。

优先读取：

- 当前 diff hunk 和相邻类 / 函数上下文；
- 目标符号的声明、定义和直接调用点；
- 项目现有注释风格；
- 格式细节重要时读取对应语言 reference 文件。

只有当注释会声明线程、生命周期、所有权、平台、协议、性能、外部来源、shader / RHI 行为、资产管线行为、引擎层行为或外部工具契约，而当前证据不足时，才继续搜索。

不要只为了润色措辞、添加例子或支撑泛化表述而搜索。

## 决策循环

使用最小有用循环。不要输出这个过程。

1. 判断用户要的是最终代码、patch、review、解释还是注释补全。
2. 检查改动文件、改动函数、公开入口、高风险代码、附近注释和相关文件头。
3. 为每个位置选择一种深度：无注释、短职责注释、约束注释或结构化 API 注释。
4. 添加、更新、移动或删除注释。
5. 验证结果。
6. 输出干净且足够时停止。

## 注释深度

### 无注释

当代码自解释，且不存在非显然职责、边界、副作用、fallback、所有权、生命周期、平台、性能、数据格式或外部契约信息时，优先不写注释。

通常不需要注释的情况：

- 简单 getter 或 setter；
- 显然谓词；
- 机械路径拼接；
- 单行转发 wrapper；
- 清晰的 private helper；
- 名称和返回路径足够清楚的脚本路径 helper；
- 未改动且注释仍准确的旧函数。

新增或改动函数必须被检查，但不自动需要注释。

### 短职责注释

当职责或边界无法从命名、类型、控制流、断言或测试中完全看出时，使用一句短注释。

短职责注释可以说明：

- 函数为什么存在；
- 它承担什么职责；
- 它刻意不承担什么边界；
- 它使用什么分类、fallback、归一化或解释规则。

不要写只表达以下内容的注释：

- 调用某个函数；
- 返回某个值；
- 设置某个变量；
- 遍历容器；
- 检查 `IsX`；
- 处理相关逻辑；
- 执行辅助操作；
- 承担局部 helper 职责。

好例子：

```cpp
// 合并植被描述和资源来源，并归一化为关键词检索文本。
FString BuildFoliagePatternSearchText(const FJX3FoliagePatternEntry& Pattern);

// 通过描述和资源来源关键词识别草本、花卉、灌木等非树木植被。
bool IsGrassLikeFoliagePattern(const FJX3FoliagePatternEntry& Pattern);
```

### 约束注释

当误用可能导致 bug、崩溃、错误渲染、数据丢失、资产损坏、性能回退或维护错误时，添加约束注释。

常见触发点：

- 必须遵守调用顺序；
- 线程、上下文、异步或可重入约束；
- 生命周期、所有权、释放或跨帧状态；
- 隐性副作用，例如 I/O、网络访问、GPU 提交、shader 重编译、缓存失效或全局状态写入；
- 单位、坐标空间、值域、颜色空间、编码假设或返回值哨兵；
- 非标准算法、经验阈值、魔数、位掩码、兼容路径、平台限制、驱动限制或引擎版本差异；
- 跨模块、跨语言、跨进程、资产管线、shader / RHI 或引擎层契约。

把约束放在未来读者最可能误用的位置附近。

### 结构化 API 注释

只有调用方需要精确使用信息时，才使用结构化 API 注释。

典型目标：

- 公开 API；
- 跨模块入口；
- 复杂 shader pass；
- 脚本入口；
- 高风险工具函数；
- 参数、返回值、异常、副作用或外部契约不明显的函数。

遵循项目和语言惯例：

- C / C++：Doxygen；
- C#：XML documentation；
- Java：Javadoc；
- JavaScript / TypeScript：JSDoc；
- Python：项目 docstring 风格；
- shader 或 shell：现有块注释或行注释风格。

不要机械填满所有标签。只有标签提供真实信息时，才添加 `@param`、`@return`、`@warning`、`@throws`、`@see` 或等价标签。

## 注释放置

函数级注释必须紧贴函数声明或定义的正上方。不要把函数级注释作为函数体内第一条语句。

函数体内注释只用于局部实现决策，例如分支原因、fallback 顺序、非显然状态变化，或只适用于特定代码块的约束。

同一个文件或同一组函数内不要混用风格。如果有些函数级注释放在函数名前，有些放在函数体内，除非注释确实只属于局部代码块，否则统一移动到函数前位置。

### 声明和定义分离

声明和定义分离时，把面向调用方的 API 注释放在声明处。把实现细节、fallback 顺序和局部约束放在定义处。

不要在声明和定义上重复同一条函数级注释。

### 组注释

共享边界使用组注释。

组注释必须用“以下 helper”“本组函数”“these entries”或等价章节标记清楚说明覆盖范围。不要让组注释看起来只属于紧随其后的一个函数。

## 函数规则

新增或改动函数和方法必须被检查，但不机械加注释。

只有当函数存在非显然职责、边界、约束、副作用、fallback、外部契约或维护风险时才写注释。

默认倾向注释：

- 公开 API；
- 跨模块入口；
- 复杂工具函数；
- 高风险渲染、shader、RHI、资产、构建、导入、迁移或数据转换函数；
- 有 fallback 行为、环境探测、非显然归一化或外部工具契约的函数。

默认倾向不写注释：

- 简单 getter 或 setter；
- 显然谓词；
- 机械路径拼接；
- 单行转发 wrapper；
- 名称和类型已经完整表达职责的 private helper；
- 只满足基类契约且不改变行为的 override 或 interface 实现。

lambda 或局部函数只有在职责不清、被跨边界传递、捕获复杂状态、跨异步 / 线程边界或有显著副作用时才需要注释。

不要因为本 skill 激活就扫未触碰的旧函数补注释，除非旧注释错误，或用户明确要求注释该范围。

如果用户明确要求选中范围内每个函数都有注释，添加简短、函数特定的注释。不要使用可复用模板。

## 文件头

只有当文件头能澄清长期文件职责、层级、边界或约束时，才添加或更新文件头。

新文件不自动需要文件头。项目要求文件头时，遵循现有项目风格。

有用的文件头可以说明：

- 文件拥有什么职责；
- 文件明确不拥有什么职责；
- 文件属于哪一层；
- 运行时、平台、线程、资源、数据格式或修改边界。

文件头不得提到：

- 任务阶段、计划编号、拆分批次或临时实现名；
- agent、AI、Codex 或 generated-by 工作流痕迹；
- 对话上下文、工单执行顺序或 review 轮次；
- 只说明当前任务为什么创建这个文件。

坏例子：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief Phase 6 拆分出的 JX3MigrationEditor 模块私有实现。
 */
```

好例子：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief 提供迁移编辑器的贴图与植被辅助逻辑，不承担资产写回和导入调度。
 */
```

## 注释坏味道

删除或重写这些坏味道。它们通常意味着注释是为了满足规则而存在，不是为了帮助维护。

### 模板化职责注释

不要在多个函数上重复可互换的职责注释。

坏例子：

- “承担当前模块内的局部辅助职责。”
- “处理相关逻辑。”
- “执行辅助操作。”
- “封装通用流程。”
- “用于完成当前功能。”
- “提供基础能力。”
- “完成必要处理。”

如果同一条注释可以不改任何字就贴到多个无关函数上，它没有有用信息。

### 名称翻译注释

不要把函数名、参数名或返回值翻译成自然语言。

坏例子：

- `RepoRoot()` 写“获取仓库根目录。”
- `PluginBaseDir()` 写“获取插件基础目录。”
- `ImportMapPlanScriptPath()` 写“定位 ImportMap plan 构建脚本。”
- `IsValid()` 写“判断是否有效。”
- `SetEnabled()` 写“设置启用状态。”

只有当路径来源、fallback 策略、运行位置、外部契约或误用风险不明显时，才给这些函数写注释。

### 满足规则型注释

不要因为规则说函数被改过，就添加注释。

要求执行的是判断是否需要注释，不是强制给每个改动函数套注释。

如果规则或用户要求每个函数都要注释，每条注释仍必须描述该函数独有的信息。

### 重复边界注释

共享边界只在最有用的位置说一次。

如果相邻 helper 共享“Python 解释源格式；C++ 消费 JSON / plan output”这类规则，优先放在：

- 文件头；
- 组注释；
- 公开入口；
- 公共 helper；
- 最可能误用的位置。

不要在每个小 helper 上重复同一个架构边界。

### 粒度错误

函数注释描述函数特定信息。

不要把模块级、阶段级或架构级解释放到小 helper 上。把这些信息移动到文件头、组注释、公开入口注释或设计文档。

### 注释掩盖命名问题

如果简单函数需要注释才能理解，先检查命名。

安全且在范围内时，优先改名，而不是添加补偿性注释。

示例：如果 `WorkspaceRoot()` 会探测运行位置并 fallback，要么解释探测 / fallback 语义，要么使用能体现解析行为的名称。不要写“承担当前模块内的局部辅助职责。”

### 函数注释和函数体注释重复

不要在函数上方解释一件事，又在函数体第一条语句重复解释同一件事。

如果函数级注释已经说明职责，函数体内注释只解释更窄的局部约束、分支原因、fallback 顺序或非显然决策。

## 路径 Helper 模式

简单路径 helper 在名称和返回表达式足够清楚时，通常不需要逐个注释。

坏例子：

```cpp
// 承担当前导入模块内的局部辅助职责。
FString RepoRoot();

// 定位 ImportMap plan 构建脚本，编辑器入口只消费已规划的目标资产。
FString ImportMapPlanScriptPath();
```

更好：

```cpp
FString RepoRoot();
FString PluginBaseDir();

// 运行位置可能是宿主工程、插件测试或仓库根；优先选择包含 source_readers 的工作区根目录。
FString WorkspaceRoot();

// 以下 helper 只定位脚本入口；源格式解释、plan 构建和 JSON 输出约定由 Python 侧维护。
FString PreparationScriptPath();
FString TerrainPreparationScriptPath();
FString ImportMapPlanScriptPath();
FString ImportFoliagePlanScriptPath();
```

## 更新已有注释

代码变化时，更新附近注释。

- 行为改变时，更新注释。
- 限制不再存在时，删除注释。
- 命名、类型、断言或测试现在已经表达的信息，删除冗余注释。
- 引入新约束时，把注释放在最可能误用的位置。
- 注释描述实现历史而不是当前行为时，删除或重写。

## 语言 Reference

格式细节重要时，只读相关 reference 文件：

- C / C++：`references/cpp-doxygen.md`
- C#：`references/csharp-xmldoc.md`
- Java：`references/java-javadoc.md`
- JavaScript / TypeScript：`references/js-jsdoc.md`
- Python：`references/python-google.md`
- Shader：`references/shader-comment.md`

如果 reference 文件缺失或不可访问，遵循项目现有风格和语言常见惯例。不要声称读过缺失的 reference。

## 输出契约

如果用户要求最终代码、patch、示例或注释补全：

- 只输出最终对外可消费版本；
- 不在 artifact 内放讨论、问题清单、建议、评分、备选方案、过程解释或元语言；
- 不在 artifact 内写“我会这样写”“推荐版本”“关键思路是”等短语。

如果用户要求 review：

- 输出发现和风险；
- 不把 review 推理、过程记录或内部判断写进代码注释。

如果用户要求解释、比较或规则设计：

- 正常面向用户回答；
- 生成的代码 artifact 要单独保持干净。

如果实际修改了文件，最终总结保持简短：

- 改了什么；
- 验证了什么；
- 还有什么未验证或风险。

## 验证

完成前检查：

- 每个新增或改动函数是否已判断是否需要注释；
- 每条注释是否提供代码之外的信息；
- 是否残留模板化职责注释；
- 是否有注释只翻译函数、参数、返回值或显然控制流；
- 是否有注释只是为了满足规则；
- 是否在相邻 helper 间重复共享边界；
- 是否把模块级或架构级解释放到了过小的 helper 上；
- 是否有函数级注释作为函数体第一条语句；
- 声明和定义是否重复同一条 API 注释；
- 组注释是否清楚说明覆盖范围；
- 内部工作流痕迹是否已移除；
- 文件头是否只描述长期职责和边界；
- 代码变化后，已有注释是否仍然准确；
- 最终输出是否干净、可对外消费。

## 停止规则

如果缺失事实会导致编造约束、来源、性能声明、平台限制、协议语义、shader / RHI 行为、资产管线行为或外部工具契约：

- 只写当前代码可验证的事实；
- 安全时使用保守职责注释；
- 只有缺失事实影响正确性或会误导时，才问一个最小问题。

如果用户要求添加、生成、修改或实现代码，但需求不明确且会影响行为、接口、数据格式、平台边界、文件范围或验证方式：

- 问一个最小问题；
- 不写猜测性实现。

如果注释会掩盖糟糕命名、糟糕抽象或错误行为：

- 安全且在范围内时，先修代码；
- 否则在最终总结中报告风险，不用注释掩盖。

## 压缩不变量

- 如果注释存在的唯一理由是“这个函数需要注释”，删除它。
- 如果注释不能说明该函数、边界或调用点独有的信息，删除它。
- 如果信息属于组、文件、模块或设计边界，移动到对应层级，而不是在小 helper 上重复。
- 函数级注释放在声明或定义上方，不作为函数体第一条语句。
- Artifact 只包含最终对外内容；过程讨论留在对话里。
