# Code Comment Skill — 中文标准代码注释

> 核心原则：**不是提高注释覆盖率，而是把隐藏约束显式化；不是所有代码都写注释，而是所有候选点都经过分档。**

---

## 0. 强制执行边界

本 skill 是代码输出前的**注释质量闸门**，不是可选风格建议。

### 0.0 Agent 必读入口

当本轮任务包含任何代码输出、创建、修改、重写或审查时，必须先执行以下入口协议；可以内部执行，不必把完整过程输出给用户。

```
入口协议
1. 识别本轮涉及的语言 / 文件类型。
2. 列出本轮新增或改动的候选点：文件头、公开 API、函数、类、分支、魔数、跨层约定、TODO/FIXME/HACK/WORKAROUND。
3. 对候选点执行 §2 信号扫描，给出 S / A / 0 / 文件头结论。
4. 输出代码或 patch 前执行 §8 自检。
```

禁止在未完成入口协议前直接输出代码。若本轮确实没有任何候选点，最终说明中写：

```
注释分档：本次改动无新增候选点，按 0 档未新增注释。
```

### 0.1 必须触发的场景

满足以下任意条件，必须触发本 skill：

- 用户要求添加 / 补全 / 生成注释、docstring、API 文档。
- 用户要求新建函数 / 类 / 模块 / 脚本 / shader / 工具。
- 用户要求修 Bug、重构、性能优化、改接口、改参数、改返回值。
- 用户要求改线程 / 生命周期 / 资源释放逻辑 / 序列化 / 渲染流程。
- 用户要求写测试、补测试、生成示例代码。
- 用户要求 Code Review、审查 patch、检查代码质量。

### 0.2 最小执行流程

即使用户没有提到“注释”，也必须按以下顺序执行：

```
步骤 1  检查本轮涉及文件的文件头状态（§1）
步骤 2  扫描本轮新增或改动的每个候选点（函数、类、分支、魔数、约定、标记）
步骤 3  对每个候选点逐条对照 §2 信号清单，第一个命中的信号决定档位
步骤 4  S 档 → 写结构化注释；A 档 → 写单行内联注释；0 档 → 不写，记录判断
步骤 5  执行 §8 自检清单，通过后输出
```

### 0.3 禁止跳过的理由

以下说法一律无效：

| 理由 | 驳回 |
|---|---|
| “用户没有要求注释” | 本 skill 默认触发，与用户是否提到无关 |
| “只是小改动” | 小改动也必须执行信号扫描；结果可以是 0 档 |
| “只是示例代码 / 脚本 / 测试” | 示例、脚本、测试同样触发扫描 |
| “代码很短” | 长度不影响是否需要扫描 |
| “稍后再补注释” | 输出前必须完成，不允许延迟 |
| “已有命名常量所以无需检查魔数” | 命名解释 What，S9 仍需检查 Why / 来源 / 边界后果 |

## 1. 文件头注释

文件头的目标是让读者第一眼知道：**这个文件做什么、不做什么、处在哪一层、有什么执行边界。**

本 skill 不追求无差别文件头覆盖率。新文件必须写；已有文件按风险和改动范围处理，避免小 patch 产生无关 diff。

### 1.1 执行规则

| 情况 | 操作 |
|---|---|
| 新建代码文件 | 必须在文件第一行写文件头；有 shebang 时写在 shebang 后第一行 |
| 修改已有文件，原文件已有文件头 | 必须核对；若与当前职责不符，必须更新 |
| 修改已有文件，原文件无文件头，且本轮改变模块职责 / 公开 API / 线程模型 / 生命周期 / 资源状态 / 数据格式 | 必须补充文件头 |
| 修改已有文件，原文件无文件头，但本轮只是局部小修，且项目既有风格无文件头 | 不强制补；最终说明写“文件头缺失，因本轮为局部小改未补” |
| 修改已有文件，原文件头准确 | 不修改；最终说明写“文件头已核对，无需修改” |

### 1.2 文件头必须回答的问题

> 读这个文件的开发者，第一句话就能知道“它做什么、不做什么、在哪一层”吗？

**必须包含：**

- 文件 / 模块名称。
- 职责说明：做什么，以及明确不做什么。

**如适用，还需包含：**

- 线程模型：仅渲染线程 / 仅游戏线程 / 主线程 / 线程安全。
- 主要依赖层次或边界：例如“不依赖平台 RHI，仅处理抽象描述符”。
- 平台约束：Windows only、Editor only、Shader Model、GPU API 等。
- 破坏性操作说明：删除文件、改配置、写注册表、批量迁移等脚本必须说明。

### 1.3 格式

按语言选择格式：

| 语言 | 格式 |
|---|---|
| C++ / C | `/** @file ... @brief ... */` Doxygen 块 |
| C# | `///` XML 文档注释或顶部 `//===` 区域说明 |
| Python | 模块级 `"""docstring"""` |
| JS / TS | `/** @file ... @description ... */` JSDoc 块 |
| Java | `/** ... */` Javadoc 块 |
| HLSL / GLSL / USF / USH | `//===...===` 分隔线块，含输入 / 输出 / Shader Model / Pass 约束 |
| Shell / PS1 | `#` 行注释块，含用途 + 平台 + 破坏性操作说明 |
| Batch / CMD | `REM` 或 `::`，避免中文乱码时可用英文极简头 |
| YAML / TOML / INI | 只在格式允许时写配置注释 |
| JSON | 不支持注释；如需解释，放到相邻 README 或 schema |

### 1.4 合格示例

❌ 不合格：零信息量，仅重复文件名。

```cpp
// RenderGraph.h
```

❌ 不合格：空洞描述，无职责边界。

```cpp
/** @brief 渲染图头文件。 */
```

✅ 合格：职责明确，边界清晰。

```cpp
/**
 * @file    RenderGraph.h
 * @brief   渲染图核心调度器，管理 Pass 依赖关系与 GPU 资源生命周期。
 *
 * 本文件不依赖平台 RHI，仅处理抽象资源描述符。
 * 实际 GPU 提交由 RHICommandList 层完成。
 * 非线程安全，所有调度操作须在渲染线程调用。
 */
```

✅ 合格：Python 模块。

```python
"""渲染资源管理模块，负责 GPU 缓冲区的分配、复用与销毁。

本模块不直接调用图形 API，所有底层操作委托给 RHIBackend 抽象层。
线程模型：所有公开方法须在主线程调用，内部使用锁保护共享状态。
"""
```

## 2. 档位判定：优先级信号清单

这是本 skill 的核心。**用可逐条核查的 binary 信号清单判定档位，替代“凭感觉多写注释”。**

### 2.0 执行方式

对代码中每个候选点，从 S 信号开始逐条检查：

- **第一个命中的 S 信号** → S 档，停止降级检查，写结构化注释。
- **未命中任何 S 信号，第一个命中的 A 信号** → A 档，写单行内联注释。
- **未命中任何 S / A 信号** → 0 档，保持沉默。

若同一候选点命中多个 S 信号，注释必须覆盖最高风险信号，并补充其他会影响调用方正确使用的信息。

---

### 2.1 S 档触发信号

满足任意一条 → 必须写结构化注释。

#### S1 — 调用顺序依赖

必须在 X 之前 / 之后调用，或调用方有初始化 / 销毁配对要求，或多次调用顺序影响结果。

```cpp
// 触发 S1：必须在 Initialize() 之后、Submit() 之前调用，否则状态机状态非法
void SetRenderTargets(const FRTArray& RTs);
```

#### S2 — 线程 / 执行上下文限制

仅限渲染线程 / 游戏线程 / 主线程调用，不可重入，或有 async / await / 协程边界约束。

```cpp
// 触发 S2：仅限渲染线程调用；从游戏线程调用会触发 assert
void FlushCommandList();
```

#### S3 — 生命周期 / 所有权约束

调用方负责释放，或返回对象仅在本帧有效，或跨帧保存引用会导致悬空 / UAF。

```cpp
// 触发 S3：返回的指针由调用方负责 delete；框架不回收
FMeshBatch* AllocateMeshBatch();
```

#### S4 — 隐性副作用

函数名不揭示的副作用：触发 I/O、网络请求、GPU 提交、Shader 重编译、缓存失效、资源重建、状态机跳转、写入全局状态。

```cpp
// 触发 S4：调用后触发 Shader 重编译（同步阻塞），不应在热路径中调用
bool SetMaterialParam(FName Key, float Value);
```

#### S5 — 参数有隐性单位 / 坐标系 / 值域

参数单位非显然，坐标系不显然，或值域有隐性约束：必须已归一化、必须大于 0、必须为 2 的幂等。

```cpp
// 触发 S5：radius 单位为世界空间米，不是屏幕像素；传入像素值会导致光照半径异常
float ComputeAttenuation(float radius, FVector3f pos);
```

#### S6 — 返回值有特殊语义

`false` 不代表异常，`nullptr` 是合法返回值，`-1` / `0` 是哨兵值，空集合有特定含义，返回值所有权由调用方持有。

```cpp
// 触发 S6：返回 nullptr 表示资源已被 GC 回收（不是错误），调用方须检查后再使用
UMaterialInterface* GetMaterial() const;
```

#### S7 — 非标准算法 / 论文实现 / 领域特定公式

实现路径不是标准库或引擎标准 API，使用了数学推导、空间映射、近似算法、查表、领域特定公式，或与直觉上“最简单写法”不同。

```cpp
// 触发 S7：使用 Octahedral 映射将单位球面法线压缩到 [0,1]² UV 空间
// 参考：Cigolle et al. 2014, Survey of Efficient Representations for Independent Unit Vectors
float2 EncodeNormal(float3 n);
```

#### S8 — WORKAROUND / HACK / 驱动兼容 / 历史兼容

任何为规避引擎 Bug、驱动 Bug、平台限制、历史数据格式或过期 API 而写的非常规代码，无论多短，都必须写 S 档注释。

```cpp
// WORKAROUND(leiMH): NVIDIA 535.xx UAV+SRV 同帧访问存在 race condition。
// 强制插入 barrier 绕过；535.154 驱动修复后可移除。
DeviceMemoryBarrier();
```

#### S9 — 魔数 / 位掩码 / 硬编码绑定点

生成或修改代码时，必须主动扫描本轮新增或改动范围内的字面量，不能只处理“看起来可疑”的数字。

**触发条件：**

- 数字字面量直接出现在逻辑中，且含义不显然。
- 已提取为命名常量，但名字仍无法解释数值的**来源**或**取值理由**。
- 位掩码或标志位组合。
- Shader register / binding / SV_Target 序号 / channel index。
- 文件格式幻数、协议版本号、字段偏移量。
- 阈值、容限、超时、重试次数、硬件限制、对齐边界。

**按类型写注释：**

| 魔数类型 | 典型形式 | 注释必须回答的问题 |
|---|---|---|
| 数学 / 物理常数 | `1.61803f`、`0.2126f`、`3.14159f` | 常数名称 + 所属公式或标准 + 在此代码中的具体用途 |
| 精度 / 误差容限 | `1e-5f`、`0.045f`、`1e-7` | 允许的误差范围 + 超出时会发生什么 + 数值来源 |
| 阈值 / 上下界 | `> 3`、`< 0.05f`、`>= 256` | 超出或低于时的行为 + 数值来源 |
| 位掩码 / 标志位 | `0x80000000`、`& 0x0F`、`\| 0x01` | 每个有效 bit 的含义 + 组合值语义 + 与枚举 / 宏的对应关系 |
| Shader 绑定点 | `register(b0)`、`SV_Target1`、`: TEXCOORD2` | 对应 CPU 侧绑定名称或枚举值 + 修改须同步的位置 |
| 对齐 / 填充 / 偏移 | `float _pad[3]`、`offset = 256`、`alignas(16)` | 对齐要求来源 + 不满足时的后果 |
| 计数 / 超时 / 重试 | `maxRetry = 3`、`timeout = 5000`、`poolSize = 64` | 达到上限时的行为 + 数值来源 |
| 文件格式幻数 | `0x4D42`、`"RIFF"`、`0x89504E47` | 格式名称 + 规范来源 + 在格式结构中的位置和含义 |
| 版本 / 协议号 | `version == 2`、`0x0300` | 对应格式 / 协议版本名 + 与其他版本的兼容性差异 |
| 硬件 / 平台限制 | `maxMRT = 8`、`warpSize = 32`、`tileSize = 16` | 来源 + 超出时的行为 |

**S9 不触发白名单：**

以下字面量默认不触发 S9，除非它们参与跨层协议、资源绑定、文件格式、硬件限制、特殊返回语义或行为边界改变：

- `0` / `1` 用于常规计数、布尔转换、数组首元素、普通初始化。
- `-1` 已由类型名、枚举名或 API 约定明确表达为 invalid / none / not found。
- 简单循环边界：`for (int i = 0; i < count; ++i)`。
- 标准 RGBA / XYZW 通道访问，且上下文明确。
- 常规字符串字面量：日志文本、错误消息、UI 文案。
- 测试代码中的少量 fixture 数值，且测试名已经解释场景。

**合格 vs 不合格：**

❌ 注释存在但无信息量，只重复名字。

```cpp
#define MIN_PERCEPTUAL_ROUGHNESS 0.045f  // 最小感知粗糙度
```

❌ 完全未注释，按 0 档处理了魔数。

```cpp
if (roughness < 0.045f) roughness = 0.045f;
```

✅ 回答来源 + 后果。

```cpp
// GGX NDF 在 perceptual roughness → 0 时趋于 Dirac delta，可能导致镜面高光 INF。
// 0.045 是 UE4 PBR 的经验下限，对应线性粗糙度 alpha ≈ 0.002。
#define MIN_PERCEPTUAL_ROUGHNESS 0.045f
```

#### S10 — 跨模块 / 跨层 / 跨语言约定

C++ 与 C# / Python / Lua / Blueprint / shader 之间共享的数据布局、字段顺序、枚举值映射；调用方和被调用方之间隐性的资源状态约定；违反会造成难以追查的 Bug。

```hlsl
// Slot 索引与 C++ 侧 RenderTarget 绑定顺序严格对应，修改须同步更新 C++ 枚举。
#define GBUFFER_SLOT_BASECOLOR  0
#define GBUFFER_SLOT_NORMAL     1
```

#### S11 — 本轮修改改变了行为边界

本轮改动改变了以下任意内容：错误处理语义、返回值语义、线程要求、生命周期 / 所有权、兼容性行为、数据格式、GPU 资源状态、渲染 Pass 顺序。

命中 S11 时，必须同步更新或新增注释；旧注释与新行为不一致时，必须删除或改写旧注释。

### 2.2 A 档触发信号

未命中任何 S 信号，满足任意一条 → 写一行内联注释。

#### A1 — 分支的存在理由不显然

条件表达式的 Why 无法从代码本身推断。写 Why，不写 What。

```cpp
if (StencilValue == 0) continue;
// ✅ Stencil = 0 为天空盒像素，GBuffer 无有效表面数据，不参与光照计算。
```

#### A2 — 代码看起来像 Bug 但是故意的

故意的空 case、故意不 break、故意的 NaN 传播、故意跳过某项初始化、故意的整数溢出回绕。

```cpp
case 2:
case 3:
    ApplySubsurfaceShading();  // 故意 fallthrough：Stencil 2 和 3 共享同一着色路径。
    break;
```

#### A3 — 单个外部约定

来自引擎、驱动、协议或平台的单条隐性约定，一句话能说清楚，不需要 S 档完整块。

```cpp
// Safari 16 之前 IndexedDB 在事务内同步读可能死锁，保留异步路径兼容旧版。
await db.get(key);
```

#### A4 — 非直觉的性能写法

用位运算代替除法、rsqrt 近似归一化、手动展开循环等。必须有数据或引用支撑，不能只说“更快”。

```cpp
int slot = index >> 3;  // 每帧调用约 1M 次，Profiler 证实比除法快 3%。
```

#### A5 — 短期兼容分支

显式标注何时可删除的过渡代码、版本升级后可移除的绕道，必须说明移除条件。

```cpp
if (legacyMode) ConvertV1Asset();  // 兼容旧版资产格式；迁移完成后删除此分支。
```

---

### 2.3 0 档

未命中任何 S / A 信号 → 不写注释，保持沉默。

| 场景 | 原因 |
|---|---|
| `GetName()`、`IsEmpty()`、`SetVisible()` 等名字自解释函数 | 注释只能重复名字，无增量信息 |
| 简单 getter / setter，无副作用、无隐性约束 | 同上 |
| 标准惯用法：普通 `for` 循环、null 检查、类型转换、标准库调用 | 经验开发者秒懂 |
| 私有 helper，公开接口已有充分注释，且实现无隐藏约束 | 实现细节由代码本身表达 |
| 注释内容只能重复函数名 / 变量名 / 语法行为 | `i++; // i 自增 1` 是反例 |
| 为满足覆盖率而生成的空洞 `@brief` / `@param` / `@return` | 无增量信息，是噪声 |

**特别禁止：** 为所有 public 方法机械生成 `@brief` / `@param` / `@return`。本 skill 反对“注释覆盖率”思维。

0 档不是跳过 skill。0 档代表“已判断，不应写注释”。当本轮没有新增注释时，在输出说明中写：

```
注释分档：本次新增/修改代码均为自解释逻辑，按 0 档未新增注释。
```

## 3. 各信号触发后的注释内容要求

S / A / 0 信号决定**是否**写注释；本节决定 S 档注释**必须写什么**。

一条注释不是写了就合格。它必须覆盖触发信号对应的关键问题。只写 What、没有回答 Why / 约束 / 来源，等同于没写。

| 触发信号 | 注释必须回答的问题 |
|---|---|
| S1 调用顺序依赖 | 必须在哪个操作之前 / 之后调用；顺序错误会导致什么；是否有销毁 / 释放配对 |
| S2 线程 / 上下文限制 | 必须在哪个线程 / 上下文调用；错误调用的后果；是否可重入；异步边界在哪里 |
| S3 生命周期 / 所有权 | 谁负责释放；对象何时失效；跨帧保存是否安全 |
| S4 隐性副作用 | 会触发哪些外部操作；是否可在热路径调用；失败时副作用是否回滚 |
| S5 参数单位 / 坐标系 / 值域 | 非显然参数的单位；坐标系；有效值域；越界行为 |
| S6 返回值特殊语义 | `null` / `false` / `-1` / `0` / 空集合分别表示什么；所有权归属；如何区分正常与错误 |
| S7 非标准算法 | 核心思路；关键假设；与标准写法的差异及原因；来源；适用范围 |
| S8 WORKAROUND / HACK | 触发条件；影响范围；移除条件；外部跟踪信息 |
| S9 魔数 / 位掩码 | 见 §2.1 S9 分类表，按对应类型填写 |
| S10 跨层约定 | 约定内容；对端位置；违反时的现象；修改时须同步更新的位置 |
| S11 行为边界改变 | 具体改变了什么；新行为约束；是否影响现有调用方；旧注释是否已更新或删除 |

若同一候选点命中多个 S 信号：

- 必须覆盖最高风险信号。
- 必须补充其他会影响调用方正确使用的信息。
- 不要求机械回答所有表格问题；避免把注释写成小论文。

**自测方法：** 写完注释后，遮住代码只读注释。注释里的信息量是否足够让调用方正确使用这段代码？如果还需要看代码才能理解约束，注释不合格。

## 4. 各档位格式规范

### 4.1 S 档格式

使用语言对应的结构化注释块：Doxygen / XML doc / Google docstring / JSDoc / Javadoc。按需填写以下标签，**每个标签都必须有实质内容，没有实质内容的标签一律不写**。

| 标签 | 何时写 |
|---|---|
| `@brief` / 首行摘要 | 始终写，但必须包含代码名字之外的增量信息 |
| `@param` | 参数有非显然值域 / 单位 / 坐标系 / 约束时 |
| `@return` | 返回值有特殊语义时 |
| `@note` | 有调用顺序、线程限制、生命周期、所有权要求时 |
| `@warning` | 误用会导致崩溃、数据破坏、GPU 状态错误、死锁、资源泄漏时 |
| `@throws` / `@exception` | 有受检异常或文档化的非受检异常时 |
| `@see` | 有配对函数、协议文档、Issue、设计文档时 |

S 档注释必须覆盖触发信号要求的关键信息。若不能回答任何 Why / 约束 / 来源 / 后果，说明它可能应该降为 A 档或 0 档。

**禁止：** `@param x x 坐标`、`@return 返回结果`、`@brief 初始化对象` 等无信息量标签。

### 4.2 A 档格式

- 单行内联注释，使用语言对应的行注释符：`//` / `#` 等。
- 紧跟在对应代码行后，或放在对应代码行正上方。
- 严禁升格为块注释：一行能说清楚的，不写成五行。
- 必须写 Why：原因 / 约束 / 例外；不写 What：代码正在做什么。

### 4.3 WORKAROUND / HACK / FIXME / TODO

任何遗留问题标记必须包含：**责任人 + 触发条件 / 现象 + 关联信息（版本号 / Issue / PR）**。

```cpp
// TODO(username):   事项描述 + 背景（Issue 编号）
// FIXME(username):  Bug 描述 + 复现条件
// HACK(username):   非常规做法的原因 + 何时可以移除的条件
// WORKAROUND(username): 外部 Bug/限制说明 + 修复版本号（已知时）
```

裸的 `// TODO`、`// FIXME`、`// HACK`、`// WORKAROUND` 一律不允许存在。

生成新代码时，除非用户明确要求，否则不输出 TODO。若必须输出，使用可追踪格式，不要写 `someone` / `unknown` / `later` 作为责任人。

## 5. 中文书写规范

- 中文与英文 / 数字之间加空格：`// 读取 GBuffer 的第 0 通道`。
- 专有名词保留英文：`// 基于 BVH 加速结构的光线求交`。
- 变量名 / 函数名保持代码原文：`// 当 maxRetries 为 0 时表示无限重试`。
- 禁止机器翻译腔：不写“该方法用于进行……的操作”。
- 用主动句、陈述句：`// 跳过天空盒像素`，不写“此处用于跳过天空盒像素的判断逻辑”。
- 注释短句化，避免一行塞入多个独立事实。
- 不在注释里堆砌“显然、简单、只是、很容易”等主观词。
- 不用中文替换代码符号、API 名、配置名、宏名。
- 不用注释掩盖糟糕命名；若能安全改名，优先改名。

## 7. 执行流程

### 7.1 生成新代码

```
1. 识别语言和输出位置。
2. 新建代码文件立即写文件头注释（§1）。
3. 列出所有新增的公开入口 / 跨文件入口 / 配置入口 / shader pass 入口 / 测试入口。
4. 对每个候选点逐条扫描 §2 信号清单，判定 S / A / 0 档。
5. 写代码时同步写入 S / A 档注释。
6. 删除所有只重复代码含义的注释。
7. 执行 §8 自检清单。
```

**各类型代码的默认起点：**

| 代码类型 | 默认处理 | 说明 |
|---|---|---|
| 新增公开 API | S 档评估 | 即使最终降为 0 档，也必须过一遍 S 信号清单 |
| 新增私有 helper | 0 档起步 | 除非命中 S/A 信号 |
| 新增脚本入口 | S 档评估 | 重点检查参数、路径、平台、破坏性操作 |
| 新增 shader pass / kernel | S 档评估 | 重点检查输入纹理、坐标空间、通道语义、Pass 顺序 |
| 新增测试 | A 档评估 | 回归测试 / synthetic 数据规则复杂时升为 S 档 |

### 7.2 修改已有代码

```
1. 找出新增 / 删除 / 修改的 hunk。
2. 检查文件头注释是否需要补充或更新（§1）。
3. 扫描 hunk 内和相邻 API 的现有注释，判断是否仍然正确。
4. 对新增逻辑执行 §2 信号扫描，判定 S / A / 0 档。
5. 添加、更新或删除注释，使注释与修改后的行为一致。
6. 不因注释需求扩大无关修改范围。
7. 执行 §8 自检清单。
```

**必须更新注释的情况（命中 S11）：** 参数含义 / 单位、返回值语义、错误处理、线程要求、所有权、生命周期、性能假设、兼容性路径、外部协议、数据格式、GPU 资源状态。

**必须删除注释的情况：** 与代码行为不一致、只是重复代码、已被命名 / 类型 / 断言替代、描述已不存在的限制、误导读者以为存在额外副作用。

### 7.3 Code Review

```
1. 检查是否缺少 S 档注释（遗漏 S1~S11 信号）。
2. 检查是否存在 A 档场景被写成大块注释（应收缩）。
3. 检查是否存在 0 档噪声注释（应删除）。
4. 检查 TODO/FIXME/HACK/WORKAROUND 是否可追踪。
5. 检查注释是否与当前代码行为一致。
6. 检查文件头策略是否符合 §1。
```

输出建议格式：

```
[comment:S] 位置：缺少 SX 信号对应的注释，建议补充……
[comment:A] 位置：可用单行替代当前块注释，建议精简……
[comment:0] 位置：注释只重复代码，建议删除……
[comment:file-header] 位置：文件头缺失或过期，建议……
```

## 8. 自检清单（输出前必须逐条核查）

```
入口协议
□ 本轮是否识别了语言 / 文件类型？
□ 本轮是否列出了候选点：文件头、公开 API、函数、类、分支、魔数、跨层约定、标记？
□ 每个候选点是否都经过 §2 信号清单扫描？

文件头
□ 新建代码文件是否已有中文文件头注释？
□ 已有文件头是否仍与职责、线程模型、平台边界一致？
□ 已有文件缺失文件头时，是否按 §1.1 判断为“必须补充”或“局部小改不补”？
□ 最终说明是否包含文件头结论？

档位判定
□ 本轮新增或改动范围内的字面量是否经过 S9 扫描？
□ 每个魔数是否按 §2.1 S9 分类表找到对应类型，并回答来源 / 边界 / 后果？
□ S9 白名单是否只用于常规自解释字面量，没有掩盖跨层约定或硬件限制？
□ 每个新增或改动的公开 API 是否经过 S1~S11 逐条核查？
□ 每个跨模块 / 跨层 / 跨语言约定是否触发 S10，已文字化？
□ 每个非标准算法 / WORKAROUND / HACK 是否触发 S7/S8，已注明来源和移除条件？

注释质量
□ 每条 S 档注释是否覆盖触发信号要求的关键信息？
□ 每条 A 档注释是否写 Why，而不是 What？
□ 是否存在只重复函数名 / 变量名 / 语法行为的注释？如有，已删除。
□ 是否存在裸的 TODO/FIXME/HACK/WORKAROUND？如有，已补充或删除。
□ 是否存在无实质内容的 @param / @return？如有，已删除。

一致性
□ 修改旧代码时，旧注释是否仍与新行为一致？不一致的已更新或删除。
□ JSON 等不支持注释的格式中，是否避免了非法注释？
□ 是否避免了因注释需求扩大无关 diff？
```

## 9. 输出要求

### 9.1 输出代码时

- 注释嵌入原始代码，保持原有缩进和结构。
- 不为增加注释而重排无关代码。
- 不为注释修改命名，除非用户要求重构，或命名修改比注释更小更安全。
- 不输出与代码行为不一致的注释。
- 不用注释掩盖糟糕命名；能安全改名时，优先改名。
- 不因文件头、S9 或自检要求扩大无关 patch 范围。
- 不把内部扫描过程长篇输出给用户；最终只保留必要分档说明。

### 9.2 输出 patch / 修改总结时

最终总结必须包含极短的“注释分档”说明。

推荐模板：

```
注释分档：
- S：为 XXX 补充结构化注释（触发 S2：渲染线程限制 / S5：参数单位 / S8：WORKAROUND）。
- A：为 XXX 增加单行注释（触发 A1：分支 Why / A4：性能写法）。
- 0：其余 getter / 简单循环按 0 档保持无注释。
- 文件头：[新建并写入 / 已补充缺失 / 已核对无需修改 / 局部小改未补]。
```

若本轮无新增注释：

```
注释分档：本次改动均为自解释逻辑，按 0 档未新增注释。文件头：已核对无需修改。
```

若本轮只是 Code Review：

```
注释审查：发现 N 个 S 档遗漏、N 个 A 档可精简点、N 个 0 档噪声注释、N 个文件头问题。
```

### 9.3 不确定时

- 不编造注释。
- 在最终说明中列为“需要确认的注释前提”。
- 可写保守注释，但只描述代码中可验证的事实，不推测隐含约束。
- 对外部协议、驱动版本、平台限制不确定时，优先标注“需确认来源”，不要伪造 Issue / PR / 文档编号。
