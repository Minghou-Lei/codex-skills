# Code Comment Skill

Role: 作为代码注释质量门，帮助 GPT-5.5 在写代码、改代码、审查代码或补注释时生成少而准、面向长期维护的中文注释。

## Goal

让代码读者不用知道本次任务背景，也能从必要注释里理解：

- 这段代码承担什么长期职责。
- 调用、修改、运行它时有哪些非显然约束。
- 哪些信息无法从命名、类型、控制流或测试中直接看出。

成功标准：保留或新增的每条注释都有可维护价值；删除只重复代码、暴露内部流程或误导读者的注释。

## Operating Mode

先判断用户要什么输出，再决定注释密度：

- 最终代码 / patch / 示例 / 注释补全：只输出最终对外版本，不暴露判断过程。
- Code review：输出问题和风险，不把 review 过程写进代码注释。
- 解释代码：可以说明为什么某类注释应写或应删，但不要把解释文本伪装成代码注释。

不要为了执行本 skill 展示清单。内部做一次轻量判断即可。

## Decision Pass

写出或修改代码前，快速过一遍：

1. 文件头：新文件是否需要说明长期职责和边界；已有文件头是否过期。
2. 函数定义：新增或改动的函数 / 方法定义是否有一句职责注释。
3. 高风险点：是否存在线程、生命周期、所有权、副作用、单位、坐标系、返回语义、跨层约定、平台限制、shader / GPU / 资源状态等非显然约束。
4. 旧注释：邻近注释是否因本次改动变成错误、过期或多余。
5. 输出形态：用户要最终代码时，最终答案是否只剩可交付版本。

只在结果需要改变代码或回答时体现判断，不输出这套过程。

## Defaults

- 注释语言用中文；代码符号、API 名、配置名、宏名、路径和协议字段保持原文。
- 优先让代码自解释；注释只补代码表达不了的意图、边界、来源、后果或约束。
- 新增或改动的函数 / 方法定义，即使逻辑自解释，也写一句简短职责注释。
- 新文件需要文件头；已有文件只在职责、边界或已有文件头过期时补充或更新。局部小改不要为了补文件头扩大无关 diff。
- 按项目既有注释风格落地；没有现成风格时使用语言惯例：C++ Doxygen、C# XML、Python docstring、JS/TS JSDoc、Java Javadoc、shader / shell 用简短块或行注释。

## Write A Comment When

写注释的判断目标是“读者是否会误用、误改或误删”。出现以下任一情况，通常需要注释：

- 调用顺序、线程 / 上下文、生命周期、所有权或资源释放有要求。
- 函数名看不出副作用，例如 I/O、网络、GPU 提交、缓存失效、状态机跳转、全局状态写入。
- 参数或返回值有非显然单位、坐标系、范围、特殊哨兵值、所有权或错误语义。
- 算法、公式、启发式、兼容路径、性能写法、魔数、位掩码或平台限制不是常规写法。
- 代码跨模块、跨语言、跨进程、跨资产管线、跨 shader / RHI / 引擎层约定。
- 本次改动让旧注释、旧边界或旧行为说明不再准确。

## High-Risk Signals

以下场景不要求大段注释，但必须确认读者能看到关键约束：

- 渲染、shader、GPU 资源、RHI、Render Graph、descriptor、barrier、queue、fence、frame lifetime。
- 引擎编辑器工具、资产迁移、批处理、文件写入、配置写入、注册表、外部进程。
- 多线程、异步、回调、事件订阅、对象池、缓存、单例、全局状态。
- 序列化格式、二进制布局、协议字段、跨语言绑定、反射、脚本暴露 API。
- 性能优化写法、非标准算法、经验阈值、平台兼容路径、驱动或版本限制。

能用命名、类型、断言或局部重构表达清楚时，优先改代码；不能表达清楚时才写注释。

## Do Not Write A Comment When

不要为了覆盖率、礼貌或形式感写注释。以下注释应删除或不写：

- 只复述函数名、变量名、语法动作或显然控制流。
- 空洞的 `@brief`、`@param`、`@return`，没有参数边界或返回语义增量。
- 解释 agent、AI、Codex、用户要求、review 轮次、任务阶段、拆分批次、临时计划、`Phase`、`Step`、“本轮改动”等内部过程。
- 把不确定信息写成事实，或编造 issue、PR、文档编号、平台限制、性能结论。
- 用注释掩盖糟糕命名；如果安全且在范围内，优先改名或收窄代码。

## File Headers

文件头只回答长期问题：文件做什么、不做什么、处在哪一层、有哪些运行或修改边界。

新文件默认写文件头。已有文件只有在下面情况才补或改：

- 原文件头缺失，且本次新增了公开入口、模块职责、跨层边界、危险副作用或长期约束。
- 原文件头存在，但与当前职责、线程模型、平台边界或数据流不一致。

不合格：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief Phase 6 拆分出的 JX3MigrationEditor 模块私有实现。
 */
```

合格：

```cpp
/**
 * @file JX3MigrationEditorModule_TextureFoliageHelpers.cpp
 * @brief 提供迁移编辑器的贴图与植被辅助逻辑，不承担资产写回和导入调度。
 */
```

## Function Comments

函数 / 方法定义的默认注释是一句话，写职责或边界，不写过程和复述。

适用范围：

- 新增或改动的自由函数、成员函数、静态函数、公开 API、私有 helper、脚本入口、测试 helper。
- lambda 或局部函数只有在职责不明显、被传递、跨异步边界、捕获复杂状态或承担副作用时才需要注释。
- override / interface 实现如果只是直接履约，可以不重复接口文档；如果改变边界、补充平台行为或有副作用，需要一句说明。

示例：

```cpp
// 合并植被描述和资源来源，并归一化为关键词检索文本。
FString BuildFoliagePatternSearchText(const FJX3FoliagePatternEntry& Pattern);

// 通过描述和资源来源关键词识别草本、花卉、灌木等非树木植被。
bool IsGrassLikeFoliagePattern(const FJX3FoliagePatternEntry& Pattern);
```

如果函数命中高风险约束，再升级为结构化注释，覆盖调用条件、参数边界、返回语义或失败后果；不要机械补全所有标签。

## Update Existing Comments

改代码时同步审查邻近注释：

- 与新行为不一致的注释要更新或删除。
- 过期限制、已不存在的副作用、已被类型 / 断言 / 测试表达清楚的信息要删除。
- 新增约束只注释在最靠近读者会误用的位置。
- 不为补注释重排无关代码，不为补文件头扩大无关 patch。

## Output

- 用户要最终代码、patch、示例或注释补全时，只给最终对外版本。
- 最终对外版本不包含讨论说明、问题清单、修改建议、评分、备选方案、解释性前后文或“我会这样写 / 建议写成 / 关键是”等元语言。
- 用户要 code review 时，按 review 输出问题；不要把 review 过程写进代码注释。
- 修改完成后的总结保持短，只说明改了什么、验证了什么、还剩什么风险。

## Stop Rules

如果缺少事实会导致注释编造约束、来源、性能结论或平台限制，先只写代码中可验证的事实；必要时问一个最小问题。

完成前做一次轻量检查：

- 新增 / 改动函数定义是否都有一句职责注释。
- 注释是否解释了代码外的信息，而不是复述代码。
- 是否清除了内部流程痕迹。
- 旧注释是否仍然准确。
- 输出是否是干净的最终版本。
